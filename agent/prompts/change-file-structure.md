# Change file structure

Use this structure exactly. Replace every placeholder with content for the active Change. Keep sections concise and testable.

## Goal

Describe the single outcome this Change should deliver.

## Scope

- List behavior, documentation, architecture, or implementation areas included in this Change.
- Keep each bullet tied to the active Change.

## Requirements

- State testable requirements using product vocabulary from `docs`.
- Include expected behavior and important boundaries.

## Acceptance Criteria

- Define observable success conditions.
- Include routes, commands, API behavior, UI states, persistence behavior, or workflow outcomes when relevant.

## Non-Goals

- List related work that is intentionally out of scope.
- Include decisions that prevent accidental scope expansion.

## Design Notes

- Record important implementation constraints, data model assumptions, or UX details.
- Link back to authoritative docs instead of repeating long explanations.

## Relevant Specs

- `docs/<path>.md`

## Verification

- Add every command needed to verify this Change.

## QA Test Cases

- List every testing scenario needed for QA to test this Change.

## Review Focus

- Call out risky or subtle areas reviewers should inspect first.

## Follow-Ups

- List useful future work that is outside this Change.
- Use `- None.` when there are no known follow-ups.
